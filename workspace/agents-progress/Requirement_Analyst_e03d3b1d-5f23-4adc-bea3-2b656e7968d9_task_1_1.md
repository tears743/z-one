## 2) 默认假设与“用户不回应时的处理策略”
> 适用于：Planner 未能在约定时间内回复或部分问题未决。

### 2.1 默认假设（可直接落地）
1. **范围默认**：仅围绕用户给定的“待确认点”产出，不主动扩写需求范围；若发现强依赖项，列为“新增待确认点”。  
2. **优先级默认**：影响范围/成本/合规最大的点默认为P0，其余P1；无法判断时默认P1。  
3. **决策默认**：没有明确决策人时，默认由 Planner 作为需求负责人确认；若涉及合规/法务/财务，默认必须升级到对应Owner。  
4. **来源默认**：优先使用一手/官方/内部权威资料；无则使用可复现的现网验证；仍无则使用公开资料并标注置信度。  
5. **网传版本默认**：**默认不作为结论依据**；最多作为“线索”进入“待验证”栏，且必须显著标注“非官方”。  
6. **不确定性默认**：凡无法用来源闭环的内容，一律标注为“待确认”，并给出影响与下一步动作。

### 2.2 不回应时的推进策略（SLA/节奏建议）
- T+0：输出《澄清问题清单 + 默认假设》并标注“等待确认截止时间”。  
- T+1（无回复）：按默认假设生成可执行草案，并将**P0待确认点**标为“阻塞项”，不做强结论。  
- T+2（仍无回复）：对非阻塞项继续拆解任务；对阻塞项提交“决策请求”给指定/默认Owner，并给出两套方案（保守/激进）及风险。  
- 任何时刻收到回复：以“变更记录”方式更新（版本号、变更点、影响面、是否需要重评审）。

## 3) 验收标准（字段要求、来源标注、网传规则、不确定性标注）
> 验收以“每条待确认点”为最小单元。建议输出为表格/清单，逐条可检查。

### 3.1 必须包含的字段（每条待确认点）
1. **ID**（唯一编号）  
2. **待确认点描述**（一句话可理解，避免口语歧义）  
3. **当前状态**（待确认/已确认/阻塞/不适用）  
4. **确认问题**（要问谁、问什么）  
5. **默认假设**（若无人回复，采用的默认结论/参数）  
6. **影响范围**（功能/数据/接口/合规/成本/排期）  
7. **风险等级**（高/中/低）+ **风险说明**  
8. **建议处理策略**（等待确认/并行验证/两案并行/先做可回滚方案）  
9. **所需来源与证据**（见3.2标注规则）  
10. **Owner/确认人**（角色或具体人）+ **截止时间**  
11. **验收口径**（确认后如何判定“已解决”）

> 缺任一字段＝不通过验收。

### 3.2 来源标注方式（必须可追溯）
- 每条待确认点至少给出一个“来源条目”，格式建议：  
  - **[来源类型] 标题/描述 — 链接/路径 — 版本/日期 — 关键摘录/页码/截图编号**  
- 来源类型枚举（建议）：  
  - 内部文档 / 会议纪要 / 现网验证 / 代码仓库 / 配置与日志 / 官方公告 / 第三方报告 / 网传  
- 若为“现网验证”，必须补充：环境（生产/预发/测试）、时间、账号/权限、复现步骤、关键结果（截图/录屏/日志）。

### 3.3 是否允许网传版本及标注要求
- **默认：不允许作为结论依据**。  
- 若 Planner 明确允许引用网传：  
  1. 来源类型必须标为 **[网传]**  
  2. 必须加注：**“非官方/未证实，仅作线索”**  
  3. 必须给出“验证计划”（如何用一手来源证伪/证实）  
  4. 结论字段不得写成确定语气（见3.4）

### 3.4 不确定性如何标注（强制规则）
- 文本层级必须使用以下标签之一：  
  - **已确认**：有权威来源闭环（或决策人明确确认）  
  - **待确认**：缺关键证据/无人拍板  
  - **推测**：基于有限证据推断，必须附“推测依据”  
  - **存疑**：存在冲突证据，必须列出冲突点  
- 同时给出**置信度**（高/中/低）或（0-100% 任选其一，默认高/中/低）。  
- 任何“推测/存疑/待确认”必须包含：**下一步动作**（找谁、要什么证据、预计何时）。
